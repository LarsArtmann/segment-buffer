# Performance baseline: v0.1.0 vs v0.2.0 (HEAD of `master`)

**Captured:** 2026-07-19
**Method:** `git worktree add ../sb-baseline v0.1.0`, then `cargo bench --bench <name> -- --warm-up-time 1 --measurement-time 3` in each worktree. Same machine, same Nix toolchain (cargo 1.96.2), back-to-back runs.
**Caveat:** single machine, short measurement window, no statistical regression tooling (criterion's noise ratio is not computed here). Numbers are indicative of direction and order-of-magnitude, not publication-grade. Re-run on your own hardware before quoting.

## Results

| Benchmark               | v0.1.0 (median) | HEAD (median) | Delta           | Verdict                                       |
| ----------------------- | --------------- | ------------- | --------------- | --------------------------------------------- |
| `append/batch_1`        | 17.8 µs         | 23.5 µs       | **+32% slower** | envelope write path dominates at tiny batches |
| `append/batch_100`      | 29.8 µs         | 49.2 µs       | **+65% slower** | envelope + new recovery bookkeeping           |
| `append/batch_1000`     | 135.1 µs        | 214.9 µs      | **+59% slower** | same                                          |
| `append/batch_10000`    | 1.74 ms         | 1.94 ms       | +11% slower     | amortized; gap closes as batch grows          |
| `recover/100_segments`  | 336.4 µs        | 205.9 µs      | **-39% faster** | recovery refactor paid off                    |
| `recover/1000_segments` | 4.85 ms         | 2.66 ms       | **-45% faster** | same                                          |

## Interpretation

- **Append latency regressed at small batch sizes** by 30–65%, narrowing to ~11% at the largest batch. The v0.2.0 envelope (`SBF1` magic + version + reserved bytes prepended to every segment) adds work per write, and the v0.2.0 `stats()`/`BufferStats` bookkeeping adds a small amount of work per `append`. This is the cost paid for the envelope's legacy-detection contract (false-positive rate 2⁻³² → 2⁻⁵⁶) and the atomic-snapshot API.
- **Recovery latency improved by ~40–45%** across both segment counts. The v0.2.0 recovery refactor (single-pass scan, simpler state machine) is the cause; this was the headline v0.2.0 perf goal and it landed.
- **Net for typical monitor365-shaped workloads** (large batches, frequent reads after a crash): roughly break-even or better, because the recover speedup dominates total time on cold starts while the append regression is hidden behind batch amortisation.
- **For tiny-batch, high-frequency writers** (1 event per flush, many flushes per second): append throughput is materially worse. If this is your workload, stay on `=0.1.0` and file an issue so we can prioritise the hot-path work in v0.4.0 (see [TODO_LIST.md](../../TODO_LIST.md) "Performance" — `read_from` clones, `AtomicU64` for `approx_disk_bytes`, `scan_segments()` cache).

## What this is NOT

- Not a benchmark of the encryption feature (all benches run with `--features=""`; AES-GCM cost is paid only when `cipher` is `Some`).
- Not a statistical comparison — there is no `criterion` regression report here, just medians from one run each.
- Not a reason to revert v0.2.0: the v0.2.0 envelope, opaque `CipherError`, and `stats()` API are the point of v0.2.0; the append cost is the price.

## Reproducing

```sh
git worktree add ../sb-baseline v0.1.0
( cd ../sb-baseline && cargo bench --bench bench_append -- --warm-up-time 1 --measurement-time 3 )
( cd ../sb-baseline && cargo bench --bench bench_recover -- --warm-up-time 1 --measurement-time 3 )
cargo bench --bench bench_append -- --warm-up-time 1 --measurement-time 3
cargo bench --bench bench_recover -- --warm-up-time 1 --measurement-time 3
git worktree remove ../sb-baseline
```
