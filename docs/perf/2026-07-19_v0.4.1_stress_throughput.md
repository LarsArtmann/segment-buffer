# Performance baseline: v0.4.1 stress throughput

**Captured:** 2026-07-19
**Method:** `cargo test --lib --features encryption --release stress_8_writers_2_readers_throughput -- --nocapture`. Same machine, cargo 1.96.2 via Nix, release profile, single back-to-back run.
**Caveat:** single machine, single run, no criterion statistical window — the stress test is a fixed-workload throughput probe, not a regression benchmark. Treat the number as order-of-magnitude; for precise deltas, re-run on quiescent hardware.

## Result

| Workload                              | Throughput             | Latency (per event) | Items observed by readers |
| ------------------------------------- | ---------------------- | ------------------- | ------------------------- |
| 8 writers × 2 readers × 80,000 events | **396,790 events/sec** | 2.52 µs/event       | 158,701 (~1.98× expected) |

## Workload shape

- 8 writer threads each append 10,000 events (80,000 total).
- 2 reader threads drain the buffer via `read_from(start, 1024)` in a tight loop until all 80,000 events have been observed by **both** readers.
- `FlushPolicy::Manual` — no I/O on the hot path; the test exercises the in-memory coordination cost under contention.
- Final assertion: a single read after all writers join observes exactly 80,000 events with contiguous sequence numbers 0..=79999.

## Interpretation

- **The ~2× reader observation count is expected and harmless.** Two concurrent readers racing on the same read cursor will both observe a subset of items they didn't "own"; the test verifies correctness via the final post-join full read, not the live counter. Do not treat the 158,701 number as a throughput denominator; it is reported only to prove the readers were active.
- **~400k events/sec under 8-writer mutex contention** is the v0.4.1 baseline for the in-memory append + read path. Future changes to the hot path (`append`, `append_all`, `read_from`, the re-entrancy guard, the mutex scope) should be compared against this number.
- **The re-entrancy guard adds one `AtomicBool::compare_exchange` per locking method.** The cost is in the noise of this measurement (the test does not call `for_each_from`, so the guard is never actually set; only the load-acquire check fires).

## What changed since v0.1.0 / v0.2.0

The earlier [v0.1.0-vs-v0.2.0](./2026-07-19_v0.1.0-vs-v0.2.0.md) baseline covers the append/recover microbenchmarks. This stress baseline is complementary: it measures the **concurrent** path, which the microbenchmarks do not exercise.

Known hot-path changes between v0.2.0 and v0.4.1 that affect this number:

- v0.3.0: `AtomicU64` for `approx_disk_bytes` (removed a lock acquisition from `stats()`).
- v0.4.0: `FlushPolicy` enum replaced two separate config fields (no measurable hot-path cost; pure refactor).
- v0.4.0: `for_each_from` lending iterator added (not exercised by this stress test).
- v0.4.1: `for_each_from` re-entrancy guard (`AtomicBool`) — negligible cost when the guard is not set.
- v0.4.1: `append_all` batch primitive — not exercised by this stress test (it uses per-item `append`).

## What this is NOT

- Not a regression benchmark. There is no v0.1.0 stress baseline to compare against; the v0.1.0 test suite did not include a multi-thread throughput test.
- Not a latency distribution. The 2.52 µs/event figure is total wall-time divided by event count, not a p50/p99 latency. p50/p99 latency under load is a deferred item — see [TODO_LIST.md](../../TODO_LIST.md) "Trust depth" section.
- Not encryption-aware. The test runs with `--features encryption` but does not configure a cipher, so the cipher path is not exercised.

## Reproduction

```bash
cargo test --lib --features encryption --release stress_8_writers_2_readers_throughput -- --nocapture
```

Expect ±10–20% variance on a typical laptop due to thermal/frequency scaling and background load.
